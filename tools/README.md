# tools

В этой папке лежат инструменты разработчика и локальной подготовки
окружения при работе с репозиторием.

Это важно понимать сразу: `tools/` — не пользовательская точка входа
для установки готового релиза. Для конечного пользователя используются
package-local скрипты `install-addin.ps1`, которые лежат внутри
add-in-пакетов.

---

## Что находится в папке

### `RevitAPI/`

Локальные reference DLL для разработки и сборки add-in-проектов.

В папке хранятся опорные наборы API для граничных веток проекта:

- `2022` — для ветки Revit `2022–2024` / `net48`;
- `2026` — для ветки Revit `2025–2026` / `net8`.

Эти файлы лежат в репозитории для удобства разработки:
не нужно каждый раз вручную собирать окружение с нуля.

При необходимости их можно заменить на DLL из вашей установленной
версии Revit или из того окружения, под которое вы собираете проект.

Важно: DLL в `tools/RevitAPI/` являются сторонними файлами Autodesk
и не покрываются MIT-лицензией этого репозитория.
Перед их публичным перераспространением или переупаковкой отдельно
проверьте применимые условия Autodesk.
Краткая памятка по этому поводу лежит в `../THIRD-PARTY-NOTICES.md`.
Они нужны для локальной разработки и сборки из репозитория.

### `install-addin.ps1`

Локальный PowerShell-скрипт для разработчика.

Он устанавливает собранный add-in из текущего репозитория
в пользовательскую папку Revit:

- для Revit 2022–2024 берёт сборку `ExportIfc.RevitAddin.Net48`;
- для Revit 2025–2026 берёт сборку `ExportIfc.RevitAddin.Net8`;
- копирует весь build-output в `%APPDATA%\Autodesk\Revit\Addins\<Year>\<AddinName>\`;
- создаёт или обновляет файл манифеста `<AddinName>.addin`
  в `%APPDATA%\Autodesk\Revit\Addins\<Year>\`.

Иными словами, это быстрый способ переустановить локальную сборку
для разработки без ручного копирования DLL и манифестов.

---

## Когда использовать

Используйте `tools/install-addin.ps1`, когда нужно:

- быстро поставить add-in из локального репозитория;
- проверить Debug/Release-сборку без ручного копирования файлов;
- переустановить локальную dev-сборку после изменений.

Это обычный рабочий сценарий разработчика.

---

## Когда не использовать

Не используйте этот скрипт как основную инструкцию для конечного
пользователя, если add-in ставится из готового release-пакета.

В release-пакетах лежат свои package-local `install-addin.ps1`,
рассчитанные на установку из содержимого архива без привязки
к структуре репозитория.

---

## Требования

Перед запуском скрипта должно быть выполнено следующее:

- проект add-in уже собран;
- сборка выполнена в конфигурации `x64`;
- выбрана корректная конфигурация (`Debug` или `Release`);
- на машине установлена соответствующая версия Revit.

Скрипт ожидает build-output в одной из папок:

- `src/ExportIfc.RevitAddin.Net48/bin/x64/<Configuration>/`;
- `src/ExportIfc.RevitAddin.Net8/bin/x64/<Configuration>/`.

Если нужной сборки нет, скрипт завершится ошибкой.
И это полезная ошибка: она обычно сразу показывает,
что проблема не в установке, а в отсутствии нужного build-output.

---

## Параметры

### `-Year`

Обязательный параметр.

Поддерживаемые значения:

- `2022`
- `2023`
- `2024`
- `2025`
- `2026`

По году скрипт сам выбирает нужную ветку add-in:

- `2022–2024` → `ExportIfc.RevitAddin.Net48`;
- `2025–2026` → `ExportIfc.RevitAddin.Net8`.

### `-Configuration`

Необязательный параметр.

Поддерживаемые значения:

- `Debug`
- `Release`

По умолчанию используется `Debug`.

### `-AddinName`

Необязательный параметр.

Это имя папки add-in и имя `.addin`-манифеста.
По умолчанию используется `ExportIFCFromRevit-CSharp`.

Скрипт работает именно с тем именем, которое передано в `-AddinName`.
Это означает, что он будет:

- очищать папку `%APPDATA%\Autodesk\Revit\Addins\<Year>\<AddinName>\`;
- создавать или обновлять файл
  `%APPDATA%\Autodesk\Revit\Addins\<Year>\<AddinName>.addin`.

Поэтому без необходимости не используйте в `-AddinName` имя чужого add-in.

---

## Примеры запуска

Из корня репозитория:

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\install-addin.ps1 -Year 2024 -Configuration Debug
```

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\install-addin.ps1 -Year 2025 -Configuration Release
```

Пример с переопределением имени add-in:

```powershell
powershell -ExecutionPolicy Bypass -File .\tools\install-addin.ps1 -Year 2026 -Configuration Debug -AddinName ExportIFCFromRevit-CSharp
```

---

## Что делает скрипт при установке

Для выбранного года Revit скрипт:

1. определяет нужный add-in-проект и основную DLL;
2. проверяет наличие локальной сборки;
3. создаёт папку `%APPDATA%\Autodesk\Revit\Addins\<Year>\`, если её ещё нет;
4. очищает целевую папку add-in, чтобы не смешивать файлы разных веток;
5. копирует весь build-output;
6. создаёт `.addin`-манифест с указанием пути к установленной DLL.

---

## Результат

После успешного запуска add-in устанавливается в стандартную
пользовательскую папку Revit и становится доступен при старте
соответствующей версии Revit.

После установки один раз вручную откройте нужную версию Revit и,
если появится окно подтверждения загрузки add-in, нажмите
`Загружать всегда`. Это нужно сделать до первого batch-запуска,
иначе процесс может остановиться на таком окне.

Если скрипт не находит build-output, это обычно означает одно из двух:

- проект ещё не собран;
- проект собран не в `x64` или не в той конфигурации.

## Связь с release-моделью и документацией

Нужно различать два сценария.

### 1. Локальная разработка

Используется `tools/install-addin.ps1`.

Он работает от структуры репозитория и ставит локально собранный add-in.
Это сценарий разработчика, а не конечного пользователя.

### 2. Установка готового релиза

Используется package-local скрипт из add-in-пакета.
Он работает от содержимого архива и не требует знания внутренней
структуры репозитория.

Именно add-in-пакеты являются пользовательским сценарием распространения,
а папка `tools/` относится к dev-инфраструктуре проекта.

Package-local `install-addin.ps1` для release-пакетов хранится
в `_release_templates/addin/...`, а в сам опубликованный пакет
попадает уже через `.github/workflows/release.yml`.

### 3. Где читать подробности

- [основной manual](../_docs/ExportIFC_manual.md);
- [README репозитория](../README.md);
- [README по настройкам](../_settings/README.md).

Если меняется логика установки add-in или роли release-пакетов,
этот README нужно поддерживать согласованным с основным manual.
Если меняется package-local сценарий опубликованных add-in-пакетов,
нужно синхронно проверять ещё и `_release_templates/`, и `release.yml`.
